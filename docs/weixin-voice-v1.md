# M叽微信语音与多用户音色 v1

## 目标

在不影响现有文字聊天、主动消息、角色承诺和扣费逻辑的前提下，为 M叽增加真正的微信语音气泡发送能力。

第一阶段使用硅基流动 TTS，但实现必须保留供应商抽象层，后续可替换为 Fish Audio、GPT-SoVITS 或其他 TTS 服务。

## 核心原则

- 不直接修改 `main`
- TTS API Key 只从本机环境变量读取，不写入数据库、日志或前端
- 语音测试必须指定完整用户 UUID 和机器人账号 ID
- 测试失败不得重复发送文字、不得误扣额度
- 不允许多账号或多用户环境自动选择目标用户
- 临时音频发送后应清理，避免长期堆积
- 语音功能默认关闭，未配置时继续使用文字回复

## 音色隔离模型

语音配置必须同时考虑：

1. 用户 UUID
2. 当前选中的角色
3. 机器人微信账号 ID
4. TTS 供应商

音色读取优先级：

1. 用户对当前角色的专属音色覆盖
2. 当前角色的默认 `voice_config`
3. 系统备用音色

禁止使用一个全局音色覆盖所有用户和角色。

## 建议配置结构

角色默认配置保存在 `characters.voice_config`：

```json
{
  "enabled": true,
  "provider": "siliconflow",
  "model": "",
  "voice": "",
  "speed": 1,
  "gain": 0,
  "pitch": 0,
  "maxTextLength": 300
}
```

用户对角色的覆盖配置保存在当前用户角色关系的偏好或独立覆盖字段中：

```json
{
  "voice": {
    "enabled": true,
    "mode": "manual",
    "provider": "siliconflow",
    "model": "",
    "voice": "",
    "speed": 1,
    "gain": 0,
    "pitch": 0,
    "maxTextLength": 300
  }
}
```

`mode` 预留值：

- `off`：只发文字
- `manual`：仅管理员测试命令可发送语音
- `voice_only`：正式接入后只发语音
- `text_and_voice`：正式接入后文字和语音同时发送
- `occasional`：正式接入后由策略决定偶尔发送语音

v1 只实现 `off` 和 `manual`，其他模式只预留，不接入自动聊天。

## 模块拆分

### 1. TTS 供应商层

负责：

- 接收文字和已解析的音色配置
- 调用硅基流动 TTS
- 校验返回类型和文件大小
- 输出统一的临时音频文件
- 隐藏 API Key

### 2. 音色解析层

负责：

- 读取用户当前角色
- 合并用户覆盖、角色默认和系统备用配置
- 校验 provider、model、voice、语速和文本长度
- 返回最终可用配置

### 3. 微信语音发送层

负责：

- 上传微信语音媒体
- 构造 `voice_item`
- 发送真正的微信语音气泡
- 返回发送结果
- 失败时不回退为重复文字

### 4. 安全测试命令

测试命令必须要求：

- `MJI_ALLOW_LIVE_TESTS=1`
- `--user-id=<完整 UUID>`
- `--account-id=<机器人账号 ID>`
- `--text=<固定测试文字>`
- 可选 `--voice=<音色 ID>`

执行前打印：

- 用户 UUID
- 微信用户 ID
- 机器人账号 ID
- 当前角色
- 最终供应商、模型和音色
- 测试文字长度
- 是否会扣额度

v1 的独立发送测试不走聊天计费，不扣用户额度。

- [ ] 建立硅基流动 TTS 适配器

- [ ] 校验空响应、错误 JSON、无效音色和超时
- [ ] 添加不联网单元测试

### 阶段二：微信语音气泡发送

- [ ] 补充微信语音媒体上传类型
- [ ] 构造和发送 `voice_item`
- [ ] 只向指定账号和指定用户发送
- [ ] 发送后清理临时文件
- [ ] 失败不重复发送文字、不扣额度

### 阶段三：多用户音色覆盖

- [ ] 读取角色 `voice_config`
- [ ] 保存用户对当前角色的音色覆盖
- [ ] 按用户、角色和机器人账号解析最终音色
- [ ] 后台增加开关、模型、音色和试听入口

### 阶段四：正式聊天接入

不在 v1 首次验收范围。必须在固定文本语音发送、多账号隔离和失败保护全部通过后单独接入。

## 验收标准

- [ ] 用户发送微信语音后，M叽仍可正确理解并文字回复
- [ ] 指定用户能收到可播放的微信语音气泡
- [ ] 语音时长和播放内容正确
- [ ] 不同用户可使用不同音色
- [ ] 不同角色可使用不同默认音色
- [ ] 两个机器人账号同时运行时不会串用户、串角色或串音色
- [ ] 无效音色返回明确错误
- [ ] API 失败不发送空语音、不重复发送文字、不扣用户额度
- [ ] 临时音频文件会被清理
- [ ] `npm run check` 覆盖全部新增正式文件和测试

全部验收完成前保持 Draft，不合并。
