# 多用户扫码接入：隔离、人设与响应速度

## 为什么不会串号

多用户接入同时使用以下四层隔离：

1. **机器人账号隔离**：每次扫码确认后得到独立的 iLink 机器人账号和令牌，每个账号对应独立 `channel_account_id`；
2. **用户身份隔离**：用户身份由 `channel_account_id + provider_user_id` 唯一确定；
3. **会话隔离**：每个会话同时绑定 `user_id + user_character_id + channel_account_id`；
4. **运行时隔离**：每个机器人账号使用独立运行实例目录、会话文件、队列文件和同步状态。

消息、长期记忆、余额和人设都通过当前用户的 UUID 读取，不使用全局变量保存当前用户。

## 可视化扫码接入

运行：

```powershell
npm run admin:onboard
```

默认打开：

```text
http://127.0.0.1:8789/
```

页面可以：

- 点击生成 iLink 临时二维码；
- 显示等待扫码、已扫码待确认、接入成功和失败状态；
- 二维码约 5 分钟过期后自动刷新；
- 每次接入最多生成 3 张二维码；
- 微信确认后自动保存机器人账号；
- 展示已保存账号列表；
- 不向浏览器返回或显示机器人令牌。

正在运行的多账号监督器会自动发现新保存的账号，并启动对应独立 worker。

## 每个用户的人设

基础角色可以共用同一个全局角色模板，但每个用户拥有独立的 `user_characters` 记录。

用户专属人设保存在：

- `user_alias`：机器人如何称呼用户；
- `character_alias`：该用户看到的机器人称呼；
- `relationship_stage`：陌生、熟悉、亲密、暧昧、伴侣等；
- `preferences.personaName`：角色名称；
- `preferences.role`：身份定位；
- `preferences.personality`：性格；
- `preferences.speakingStyle`：说话方式；
- `preferences.relationship`：与用户的关系要求；
- `preferences.background`：背景设定；
- `preferences.boundaries`：边界与禁区；
- `preferences.extraPrompt`：用户专属补充指令。

每次模型回复前，系统只加载当前用户对应的人设，并与该用户自己的长期记忆一起注入。其他用户无法读取。

## 管理员人设窗口

运行：

```powershell
npm run admin:persona
```

窗口默认只监听：

```text
http://127.0.0.1:8788/
```

管理员可以选择任意已经发过消息的用户，修改机器人名字、称呼、关系阶段、身份、性格、说话方式、关系要求、背景故事、边界和额外专属指令。

保存后，从该用户下一轮回复开始生效，不影响其他用户。管理员页面只允许本机访问，不要通过端口映射、隧道或反向代理公开。

## 微信内零成本人设向导

用户直接发送：

```text
设置人设
```

系统会在进入 AI 运行层之前拦截，并使用固定本地问题逐项收集人设。

向导支持：

```text
上一步
跳过
重新开始
取消
保存
```

向导特性：

- 不调用模型；
- 不产生 API 成本；
- 不扣用户额度；
- 不写入普通聊天记录；
- 不参与长期记忆提取；
- 进度按用户保存在数据库，程序重启后可继续；
- 保存后从下一条正常聊天开始生效；
- 不同用户的进度和人设完全隔离。

## 多账号运行方式

多账号启动器会：

- 枚举已保存的机器人账号；
- 每个账号启动一个独立 worker；
- worker 使用独立实例目录；
- 所有 worker 共用 PostgreSQL，但数据库查询始终带租户、账号和用户条件；
- 单个 worker 崩溃不影响其他账号；
- 新账号保存后自动启动对应 worker。

## 响应速度

OpenAI 兼容运行层支持流式请求：

```env
MJI_API_STREAM=true
MJI_API_STREAM_CHUNK_CHARS=48
```

不需要流式时可以关闭：

```env
MJI_API_STREAM=false
```

模型生成到一个完整句子或达到合适长度后，流式模式会先把这一段发送到微信，不再等完整回复全部结束。

安全与计费规则：

- 第一段发送前先完成余额扣除；
- 最终完整回复只用于数据库保存和长期记忆提取，不会重复发送；
- 私有长期记忆区块不会流到微信；
- 若供应商不支持流式接口，会自动退回普通非流式请求；
- 日志会输出 `headersMs`、`firstTokenMs` 和 `totalMs`，用于判断延迟发生在网络、模型还是完整生成阶段。

## 上线前必须通过的测试

- 两个用户同时发消息，回复不能互换；
- 两个用户设置完全相反的人设，回复风格不能串；
- 一个用户修改称呼，另一个用户不受影响；
- 两个用户同时写长期记忆，查询结果按用户分开；
- 一个机器人账号重新登录，不影响其他账号；
- 重启全部 worker 后，各用户仍恢复自己的会话、人设、记忆和余额；
- 新二维码确认后账号自动保存，且多账号监督器自动启动；
- 浏览器和日志不得输出 `bot_token`；
- 向导完成前后余额不变化；
- 向导问答不能进入普通聊天记录或长期记忆；
- 流式回复不得泄漏长期记忆机器区块；
- 流式回复只扣一次额度，最终回复不得重复发送。
