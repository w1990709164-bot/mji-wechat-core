# 程序重启后的聊天上下文恢复

M叽微信版会把聊天消息保存在 PostgreSQL 的 `messages` 表中。

本功能在某个微信用户于本次进程中第一次发消息时：

1. 根据该用户当前 `conversation_id` 查询数据库；
2. 读取最近的用户与 M叽文本消息；
3. 排除刚刚收到、已经落库的本次新消息；
4. 将此前对话作为恢复上下文注入模型；
5. 后续消息继续使用运行时内存上下文，不再重复查询和注入。

环境变量：

```env
MJI_DB_HISTORY_MESSAGES=30
```

默认恢复最近 30 条消息，最大允许 100 条。这里的“条”包含用户消息和 M叽回复，不是 30 轮。

运行日志示例：

```text
[mji] restored history user=<user-id> conversation=<conversation-id> messages=18
```

验证方式：

1. 与 M叽聊一个明确事实，例如“我今天买了一把蓝色雨伞”；
2. 等待 M叽正常回复；
3. 停止并重新启动程序；
4. 再问“我刚才买了什么颜色的雨伞？”；
5. 终端应出现 `restored history` 日志，M叽应能根据数据库里的旧消息回答。

这属于短期聊天上下文恢复。长期记忆、人物档案和生活轨迹仍由 `memories`、用户档案及后续记忆提取流程负责。
